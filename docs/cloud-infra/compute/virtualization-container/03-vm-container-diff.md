## 개요
클라우드 환경에서 애플리케이션을 배포하고 운영하는 방식은 인프라 구조와 비용, 성능, 운영 복잡도에 큰 영향을 줍니다.
이 글에서는 가상머신(VM)과 컨테이너의 차이점과 제약 조건을 정리합니다. 

## 1. 가상머신 (Virtual Machine): 하드웨어 레벨의 가상화
가상머신은 물리적 서버의 하드웨어를 추상화하여, 하나의 물리 서버 위에서 여러 개의 독립적인 운영체제(OS)를 실행할 수 있게 해주는 기술입니다.

**아키텍처의 핵심: 하이퍼바이저 (Hypervisor)**

VM 아키텍처의 심장부는 하이퍼바이저(또는 VMM, Virtual Machine Monitor)입니다. 하이퍼바이저는 물리적 하드웨어 리소스(CPU, Memory, Storage, Network)를 논리적으로 분할하여 각 VM에 할당합니다.

> 

- Type 1 (Bare-Metal): 하드웨어 위에 직접 설치되어 하드웨어를 제어합니다. (예: VMware ESXi, KVM, Xen)

- Type 2 (Hosted): 호스트 OS 위에서 하나의 애플리케이션처럼 동작합니다. (예: VMware Workstation, Oracle VirtualBox)

**기술적 특징과 한계**
컨테이너와 비교했을 때 가상머신의 가장 큰 특징은 Guest OS의 존재입니다. 각 VM은 완전히 독립된 커널을 가진 온전한 운영체제를 구동합니다.

- 강력한 격리성(Isolation): 하드웨어 레벨에서 격리되므로, 하나의 VM이 침해되거나 커널 패닉이 발생해도 다른 VM이나 호스트 시스템에 영향을 주지 않습니다. 보안 요구사항이 엄격한 환경에 적합합니다.

- 오버헤드(Overhead): 애플리케이션을 실행하기 위해 무거운 Guest OS를 부팅하고 유지해야 하므로 시작 시간이 길고(수 분 소요), 메모리와 디스크 공간을 많이 차지합니다.

## 2. 컨테이너 (Container): OS 레벨의 가상화
컨테이너는 하드웨어가 아닌 운영체제(OS)를 추상화하는 기술입니다. 애플리케이션과 그 실행에 필요한 모든 라이브러리, 종속성을 하나의 패키지로 묶어 어느 환경에서나 동일하게 실행되도록 보장합니다.

**아키텍처의 핵심: 호스트 OS 커널 공유**

컨테이너는 하이퍼바이저를 거치지 않고, 컨테이너 런타임(예: containerd, CRI-O)을 통해 호스트 OS의 커널을 다른 컨테이너들과 공유합니다. 이것이 컨테이너가 가벼운 결정적인 이유입니다.

컨테이너의 논리적 격리를 가능하게 하는 핵심 리눅스 커널 기술은 다음과 같습니다:

- Namespaces (네임스페이스): 리소스의 가시성을 격리합니다. 각 컨테이너는 독립된 PID(프로세스 ID), NET(네트워크 인터페이스), IPC(프로세스 간 통신), MNT(마운트 포인트), UTS(호스트네임) 네임스페이스를 가지므로, 마치 독립된 OS 환경에서 동작하는 것처럼 보입니다.

- cgroups (Control Groups): 리소스 사용량을 제어합니다. 특정 컨테이너가 호스트의 CPU나 메모리를 독점하지 못하도록 물리적 자원의 할당량을 제한하고 모니터링합니다.

- UnionFS (Union File System): 여러 개의 파일 시스템을 하나로 마운트하는 기술로, 도커 이미지의 레이어(Layer) 구조를 가능하게 합니다. 변경된 부분만 새로운 레이어로 쌓기 때문에 스토리지 효율성이 높습니다.

**기술적 특징**
- 경량성 및 민첩성: Guest OS가 없기 때문에 메가바이트(MB) 단위의 작은 크기를 가지며, 밀리초(ms) 단위로 애플리케이션을 시작하고 종료할 수 있습니다.

- 높은 밀도: 동일한 하드웨어 리소스에서 VM보다 훨씬 많은 수의 애플리케이션 인스턴스를 실행할 수 있습니다.

## 3. 한눈에 보는 비교

| 비교 항목 | Virtual Machine (가상머신) | Container (컨테이너) |
| --- | --- | --- |
| 가상화 수준 | 하드웨어 (Hardware-level) | 운영체제 (OS-level) |
| 핵심 컴포넌트 | 하이퍼바이저 (Hypervisor) | 컨테이너 엔진/런타임 (예: Docker, containerd) |
| 운영체제 (OS) | 각 VM마다 독립된 Guest OS 필요 | 호스트 OS 커널 공유 |
| 크기 | 기가바이트 (GB) 단위 | 메가바이트 (MB) 단위 |
| 시작 속도 | 느림 (수 분 - OS 부팅 과정 필요) | 매우 빠름 (수십 밀리초 - 프로세스 시작) |
| 격리성 (Security) | 매우 높음 (하드웨어 레벨 격리) | 상대적으로 낮음 (커널 공유로 인한 취약점 존재 가능성) |
| 이식성 | 상대적으로 낮음 (환경에 따라 종속성 문제 발생 가능) | 매우 높음 ("Build once, Run anywhere") |

## 4. 아키텍처 설계: 언제 무엇을 선택해야 할까?
실제 프로덕션 환경에서는 두 기술이 상호 보완적으로 사용되는 경우가 많습니다. (예: AWS EC2 인스턴스(VM) 위에 EKS(Kubernetes)를 올려 컨테이너를 오케스트레이션하는 구조)

- Virtual Machine이 적합한 경우
  - 서로 다른 OS 환경이 하나의 물리 서버에 필요한 경우 (예: Linux 서버에서 Windows 애플리케이션 구동)
  - 하드웨어 자원을 독점적으로 사용해야 하는 무거운 모놀리식(Monolithic) 애플리케이션
  - 완벽한 보안 격리가 요구되는 멀티 테넌트(Multi-tenant) 환경

- Container가 적합한 경우
  - 마이크로서비스 아키텍처(MSA) 기반의 애플리케이션
  - Scale-out과 Scale-in이 빠르고 빈번하게 일어나는 환경
  - 지속적 통합/지속적 배포(CI/CD) 파이프라인 최적화
  - Kubernetes와 같은 오케스트레이션 도구를 활용한 자동화된 운영 환경 구축
